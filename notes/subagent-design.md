# Subagent报告撰写机制

## 为什么需要subagent

**原因一：防止context压缩**

写长篇报告（财报分析5章、持续跟踪4章、投资洞察5节）需要同时持有大量原始材料和生成大段文字，容易触发context压缩。压缩后报告质量下降且需要重新加载材料。

用subagent写报告，主agent的context只保存讨论脉络和关键决策，不被报告正文撑满。

**原因二：奉承偏差防火墙（财报分析尤为重要）**

主对话积累了Joshua对公司的观点和倾向。如果财报分析在主对话中执行，模型会不自觉地将分析框架、隐含风险的严重程度、竞争格局的描述，向Joshua的既有立场偏移——最危险的是第2章"本行业分析框架"，它定义了后续三章的分析方向，一旦被污染整个报告都会受影响。

Subagent只看到数据和SKILL.md，看不到主对话中的任何立场，结构性隔断了偏差来源。

## 分工

**主agent**：启动subagent、与Joshua讨论、更新研究笔记。这些是需要保持上下文连贯的交互性工作。

**subagent**：整个报告流程（数据获取+搜索+撰写），独立完成，结果写入文件。这是单向输出、不需要交互的工作。

搜索也交给subagent——主agent对搜索质量并没有有效把控机制，真正的质量把控来自Joshua的审阅和讨论中的纠正。

## subagent调用方式

使用Agent工具，subagent_type为general-purpose，model为opus。

prompt模板：
```
你是Insight金融分析系统的报告撰写agent。
任务：为{股票代码}({公司名称})生成{报告类型}。
严格按照 skills/{skill_path}/SKILL.md 中的完整工作流执行。
必读：notes/source-reliability.md（信息源可靠性分层）、notes/timeline-system.md（时间轴更新规则）。
{主agent传递的特殊注意事项，如有}
```

SKILL.md已包含完整工作流（读什么文件、怎么搜索、怎么写），subagent按SKILL.md执行即可。

## subagent的输出

- 报告文件 → `stock/{股票代码}/memory/`
- 检查点文件 → `stock/{股票代码}/cache/`
- 时间轴更新 → `stock/{股票代码}/memory/时间轴.md`

## 设计决策记录

- **搜索归属**：交给subagent。主agent现有流程中搜索质量也无有效把控，不如让subagent独立完成，搜索结果写入检查点文件可供事后追溯
- **模型选择**：opus。长上下文召回能力更强，报告质量更有保障
- **分阶段写入**：subagent有独立context，理论上可一次写完。保留分阶段机制作为fallback
- **质量保证**：subagent不自审。报告质量由Joshua审阅和后续讨论把控

## 测试记录

**2026-02-09 首次测试：600732（爱旭股份）财报分析报告**
- 数据获取（步骤1-2）由主agent完成，报告撰写交给subagent（当时的临时设计，已更新为全流程subagent）
- 后台模式不稳定：首次后台启动后卡住，resume后仍未产出报告文件
- 同步模式正常：约4.7分钟完成全部5章+后续跟踪重点（289行），报告质量正常
- 时间轴由第一次后台subagent生成，格式规范
- 主agent context基本未被占用，测试期间可正常聊天

**结论**：机制可行，优先使用同步模式。后台模式稳定性待进一步观察。

## 状态

已验证可行。后续正式使用时以同步模式为主。
