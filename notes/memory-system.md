# 记忆系统 设计笔记

## 三层存储架构：memory / cache / filings (2026-02-07)

**背景**：分析过程产生不同类型的数据，需要不同的存储策略。

**方案选择**：每个股票目录下分三层。

| 层级 | 目录 | 用途 | 组织原则 |
|------|------|------|----------|
| 长期记忆 | `memory/` | 报告、摘要、公司信息 | 围绕**重要性** |
| 短期记忆 | `cache/` | 公告摘要等临时文件 | 围绕**时效性** |
| 原始档案 | `filings/` | PDF原文件 | 不可修改的原始材料 |

**设计逻辑**：
- `memory/` 围绕重要性——经过提炼的核心分析资产，跨session长期使用
- `cache/` 围绕时效性——如"近3个月公告摘要"，过时后可删除重建
- `filings/` 保存原始PDF，需要时可重新提取，是数据的"源头"

数据流向：filings（原始）→ memory（提炼）→ 报告（输出）。cache是这个过程中的临时产物。

## 研究笔记机制设计 (2026-02-07)

**背景**：Claude每次新session都失去上下文，需要跨session延续研究思路的机制。

**方案选择**：`stock/{股票代码}/memory/研究笔记.md`——按时间倒序追加的结构化笔记。

**核心设计决策**：
- **时间倒序**：最新的判断在最前面，Claude快速获取最新状态
- **状态追踪**：待验证 / ✅ 已验证 / ❌ 已否定 / 仍未知——明确每个问题的进展（由之前的Claude在讨论中提出的方案）
- **触发时机**：讨论产生洞察时追加；投资洞察完成后自动提取
- **阅读时机**：开始任何分析工作前先读，恢复上下文

**当前状态**：机制已在CLAUDE.md中定义，已为300803、688082、601138三只股票创建研究笔记。

## 项目开发笔记系统 (2026-02-07)

**背景**：研究笔记解决的是"股票分析"的跨session连续性，但项目开发本身也需要类似机制——Claude需要理解设计决策的"为什么"。

**方案选择**：`notes/` 目录，索引+分主题文件。

**核心原则——渐进式加载**：
- 不在session开始时主动加载所有笔记
- CLAUDE.md中记录笔记系统的存在
- 涉及相关模块开发/讨论时，按需读取对应主题文件
- 避免无谓的token消耗

**为什么不放在 CLAUDE.md 里**：
- CLAUDE.md每次session自动加载，内容越多token消耗越大
- 设计决策的详细讨论不是每次都需要
- 独立文件可按主题组织，只加载需要的部分

**为什么不用单文件**：
- 项目有多个独立模块（数据源、报告、记忆系统等），关注点不同
- 分文件后可精确加载，不必为了看数据源设计而加载报告体系的历史

**写入时机**：
1. 手动：讨论产生有价值的设计决策时
2. 自动：完成功能开发或重大改动后
