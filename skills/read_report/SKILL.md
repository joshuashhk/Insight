# 读研报

## 概述

阅读卖方研究报告，提取增量信息，生成摘要和修改计划供讨论。不自动修改任何笔记文件——所有变更经讨论确认后再执行。

## 输入

单篇研报 PDF 路径。例：
```
/read-report /path/to/report.pdf
```

## 工作流

### 步骤1：加载上下文

1. 读取行业索引 `industry/{行业}/索引.md`
   - 如行业目录不存在，跳到步骤2（将作为新行业处理）
   - 从索引了解已有哪些主题、各主题当前状态

2. 根据研报标题和行业判断，预加载最可能相关的1-2个主题笔记
   - 只加载最相关的，不要加载全部

### 步骤2：预判（读摘要）

先读前5页左右（封面+摘要/核心观点+目录+图表目录），结合步骤1的已有笔记，判断增量价值：

- **全读**：新主题（现有笔记未覆盖的子行业/技术方向），或深度报告（单一主题的专题研究）
- **选读**：总览类报告但某章节有新数据或新视角，只读相关章节
- **跳过**：内容与已有笔记高度重复，无增量。输出跳过理由，进入步骤4讨论确认

将预判结果告知 Joshua，确认后再决定阅读范围。

### 步骤3：阅读研报

根据步骤2确定的范围阅读。使用 `skills/financial_analysis/extract_pdf_text.py` 提取PDF文本（如PDF过大），或直接阅读PDF。

阅读时带着以下问题：
- 这篇研报的**核心论点**是什么？
- 有哪些**新数据**（跟已有笔记对比）？
- 有没有**框架级变化**的信号？（用元框架判断：产业结构变了？新进入者？技术路线转换？供需关系逆转？）
- 有哪些**关键事件**需要记录到时间轴？
- 有没有**可疑内容**需要讨论验证？

### 步骤4：输出摘要和修改计划

输出两部分，格式如下：

```markdown
## 研报摘要

**来源**：{券商}《{标题}》{日期}
**核心论点**：{1-2句话}
**信息质量**：{简要评价：数据是否有明确来源、逻辑是否完整、是否有独特观点}

### 关键内容
- {要点1}
- {要点2}
- ...

### 可疑/待验证内容
- {如有，标注原因}

### 与现有笔记的矛盾点
- {这篇研报里哪些内容与已有行业判断相悖？没有矛盾写"无"，不可省略}

---

## 修改计划

### 框架更新（如有）
- [ ] {描述变更} → 影响文件：{路径}
- 变更理由：{为什么认为框架需要更新，基于哪个元框架判断}

### 数据更新
- [ ] {描述变更} → 影响文件：{路径}
- 旧值：{如已知} → 新值：{研报数据}

### 时间轴新增
- [ ] {事件描述}（{日期}）→ 影响文件：{路径}

### 新主题（如有）
- [ ] 建议新建：{主题名称}，理由：{为什么现有主题无法覆盖}

### 无需操作
- {列出研报中已有/重复/无增量价值的内容，说明为什么跳过}
```

### 步骤5：讨论

等待 Joshua 审阅摘要和修改计划。可能的结果：
- **采纳**：执行修改计划中确认的条目
- **修改**：调整后执行
- **看原文**：Joshua 需要查看研报原文中某个具体部分再决定
- **丢弃**：这篇研报没有增量价值

### 步骤6：执行更新（讨论确认后）

**笔记目录结构规范**（归类依据）：

| 目录 | 放什么 | 判断标准 |
|------|--------|---------|
| `分析框架/` | 结构性分析 | 跨时间有效，不随数据更新而失效：商业模式、产业链特征、技术路线对比、分析方法论 |
| `行业数据/` | 市场数据 | 周期性更新，有明确时间戳：市场规模、区域市场（中国/美国/欧洲）、竞争格局、供需数据 |
| `时间轴.md` | 时效性事件 | 时间点明确、未来会被新事件覆盖：政策发布、产品发布、重大订单、季度数据快照、项目进度 |

归类原则：**同一内容放错目录比缺失更有害**——数据混入分析框架会让框架文件快速膨胀且失真；事件混入行业数据会模糊数据时效性。有疑问时宁可新建文件也不混入。

按确认后的修改计划更新对应文件：
- 更新数据：替换旧数据，标注来源和日期
- 更新时间轴：追加条目，标注获取日期
- 更新框架：修改分析框架文件，标注变更原因
- 新建主题：创建新目录和文件，更新索引

每次更新后标注来源：`（来源：{券商}《{标题}》{日期}）`

更新索引 `industry/{行业}/索引.md`：
- 新主题加入索引
- 已有主题更新"最近更新"时间

### 步骤7：笔记长度检查

更新完成后，检查被修改文件的大小：

| 文件大小 | 动作 |
|----------|------|
| < 4KB | 无需处理 |
| 4-8KB | 检查是否有被新数据取代的旧数据、可删除的推导过程 |
| > 8KB | 必须处理：优先**拆分**，其次**精简** |

**处理策略选择：**

- **拆分**（优先）：文件内容涵盖多个独立主题，各主题数据量均有价值 → 按主题拆为 2-3 个文件，**数据不删除**，更新索引
- **精简**（次选）：文件主题单一，内容存在真正冗余（重复测算/已被取代的旧数据/推导过程）→ 压缩，目标 4KB 以内

**精简规则（优先级从高到低）：**
1. 删除被新数据取代的旧数据（只保留最新值）
2. 删除逐季度/逐月明细表，只保留趋势结论
3. 多个券商的重叠测算只保留一组
4. 详细列表压缩为一句话概述（如"各家海外工厂详情"→"金风/远景/三一/明阳已布局海外产能"）
5. 论证推导过程只保留结论

**必须保留的内容：** 结论性判断、最新核心数据、关键逻辑链条、来源标注。

笔记是索引+结论，不是研报副本。需要细节时回原文查。

## 元框架参考

以下框架分两类：**信息源校准**（读研报之前先戴上的眼镜）和**行业分析**（判断增量信息的层级）。

### 信息源校准

| 元框架 | 作用 | 应用方式 |
|--------|------|----------|
| 卖方立场偏差 | 校准信息源的激励结构 | 结论系统性偏乐观，数据比结论可靠，预测需打折；信数据不信判断 |

### 行业分析

| 元框架 | 信号 | 如果触发 |
|--------|------|----------|
| 波特五力 | 新进入者、供应商/买方议价权变化、替代品出现 | 框架更新 |
| 边际分析 | 供需关系逆转、成本结构变化 | 框架或数据更新 |
| 博弈论 | 竞争对手策略转变、合作关系变化 | 框架更新 |
| 产业生命周期 | 行业阶段跳转信号（如从导入期进入成长期） | 框架更新 |
| 技术扩散曲线 | 渗透率拐点、主流技术路线切换 | 框架更新 |
| 创造性破坏 | 新技术使旧商业模式失效 | 框架更新 |

大部分研报只会触发数据更新和时间轴新增。框架更新应该很少——如果频繁触发，说明对框架的理解还不够稳定，需要讨论。
